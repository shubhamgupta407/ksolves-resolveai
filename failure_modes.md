# Failure Mode Analysis — ResolveAI

ResolveAI follows a hybrid decision pipeline using rule-based routing, TF-IDF similarity scoring, MiniLM semantic matching, confidence thresholds, LLM fallback, and human escalation.  
Because the platform handles live support tickets, failure recovery is built directly into the workflow.

---

## 1. Low Confidence Intent Detection

### Scenario  
A ticket does not closely match known intents, or the confidence score generated by the local pipeline is below the automation threshold.

### Example  
“Payment issue but not sure what happened after using coupon and wallet.”

### System Handling  
- Rule engine checked first  
- TF-IDF + MiniLM semantic scores evaluated  
- If confidence is below threshold, auto-resolution is skipped  
- Ticket routed to LLM fallback for deeper reasoning  
- If still unclear, sent to escalation queue  

### Outcome  
Uncertain cases are never force-resolved incorrectly.

---

## 2. LLM Fallback Failure / Timeout

### Scenario  
External LLM provider is slow, unavailable, rate-limited, or returns malformed output.

### System Handling  
- Timeout protection prevents request hanging  
- Local pipeline result remains available  
- Ticket is moved to Human Escalation Queue  
- Original ticket text, scores, and trace data are preserved  
- Failure logged in audit history  

### Outcome  
No request is lost even if third-party AI fails.

---

## 3. Concurrent Batch Ticket Load

### Scenario  
Multiple tickets arrive together (demo batch of 20 or real traffic spike).

### System Handling  
- Backend processes requests asynchronously  
- Each ticket handled independently in parallel  
- One slow ticket does not block others  
- Results merged into batch summary dashboard  
- Failed items marked separately for retry/escalation  

### Outcome  
Platform remains responsive during simultaneous requests.

---

## 4. Wrong Auto-Resolution Prevention

### Scenario  
A ticket gets moderate confidence but not enough for safe automation.

### System Handling  
- Confidence threshold acts as safety gate  
- Only high-confidence intents receive automated resolution  
- Borderline scores routed upward instead of auto-solving  
- Human reviewer can inspect confidence + model used  

### Outcome  
Accuracy prioritized over aggressive automation.

---

## 5. Invalid or Empty Ticket Input

### Scenario  
Incoming payload has blank text, missing fields, or broken JSON.

### System Handling  
- Request validation before pipeline start  
- Invalid payload rejected gracefully  
- Proper error message returned  
- Other live tickets continue processing normally  

### Outcome  
Bad input cannot crash the system.

---

## 6. Audit & Recovery Visibility

### Scenario  
Admin needs to know why a ticket was automated, escalated, or failed.

### System Handling  
Every ticket stores:

- Input query  
- Detected intent  
- Confidence score  
- Model used (local / fallback)  
- Final action taken  
- Timestamp  

### Outcome  
Complete traceability for enterprise review.

---

## Final Reliability Principle

ResolveAI uses local intelligence first, fallback intelligence second, and human judgment last.  
If confidence is low or systems fail, the platform degrades safely instead of making risky decisions.
